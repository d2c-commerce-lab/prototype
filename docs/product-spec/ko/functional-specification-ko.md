# D2C 커머스 프로토타입 기능 명세서

## 문서 목적

> [!NOTE]
> 본 문서는 프로토타입의 기능 범위와 데이터 생성 기준을 정의하는 기준 문서다.  
> 구현과 데이터 엔지니어링 설계 전반에서 공통 참조 문서로 사용된다.

본 문서는 D2C 커머스 웹 프로토타입의 기능 범위와 구현 기준을 정의한다. 이 프로토타입은 완성형 상용 서비스를 목표로 하지 않는다. 핵심 목적은 D2C 커머스 환경에서 실제로 발생할 법한 사용자 행동과 비즈니스 상태 변화를 재현하고, 그 과정에서 운영계 데이터와 이벤트 로그를 안정적으로 생성하는 데 있다.

즉, 이 문서는 단순한 웹 애플리케이션 명세서가 아니다. 이후 데이터 파이프라인 구축, 데이터 품질 검증, 모니터링, BI 설계를 수행하기 위한 원천 서비스가 어떤 기능과 데이터 생성 구조를 가져야 하는지를 규정하는 기준 문서다. 따라서 프론트엔드와 백엔드 구현, 데이터베이스 스키마 설계, 이벤트 로그 설계, 나아가 후속 데이터 엔지니어링 작업의 출발점으로 활용된다.

## 프로토타입이 지향하는 서비스 모습

본 프로토타입은 라이프스타일 및 업무 보조 성격의 상품을 판매하는 가상의 D2C 커머스 플랫폼을 전제로 한다. 상품군은 데스크 셋업 액세서리, 생산성 도구, 오피스 라이프스타일 용품, 소형 업무 관련 아이템 등으로 가정한다. 이러한 도메인은 상품 탐색, 장바구니, 주문, 쿠폰, 리뷰와 같은 전형적인 커머스 행동을 자연스럽게 포함하면서도, 프로토타입 수준에서 과도한 복잡성을 유발하지 않는다는 점에서 적절하다.

이 서비스는 화면 완성도나 상용 운영 수준의 복잡한 예외 처리보다, 핵심 사용자 흐름을 끊김 없이 수행할 수 있고 그 과정이 데이터로 남는 구조를 갖추는 데 초점을 둔다. 따라서 본 프로토타입은 보기 좋은 서비스라기보다 분석 가능한 데이터를 만들어내는 서비스에 가깝다.

## 구현 대상과 제외 범위

> [!IMPORTANT]
> 이번 구현은 핵심 구매 흐름과 데이터 생성에 직접 필요한 기능만 포함한다.  
> 상용 운영에 필요한 부가 기능과 고도화 요소는 이번 범위에서 제외한다.

이번 프로토타입에서 구현해야 하는 범위는 명확하다. 사용자는 회원가입 또는 로그인을 통해 서비스에 진입할 수 있어야 하며, 홈 화면에서 카테고리와 상품을 탐색하고, 상품 상세를 확인한 뒤 장바구니에 상품을 담고 제거할 수 있어야 한다. 이후 체크아웃 단계로 이동해 쿠폰을 적용하고 주문을 생성하며, 결제 결과를 성공 또는 실패로 시뮬레이션할 수 있어야 한다. 주문 내역 조회와 구매 상품에 대한 리뷰 작성 역시 지원 범위에 포함된다.

반대로 실제 결제 시스템 연동, 관리자 페이지, 정교한 재고 관리, 복잡한 SKU 및 옵션 구조, 추천 시스템, 실시간 알림, 고도화된 권한 체계, 고객센터 운영 흐름, 상용 배포 수준의 운영 안정화는 이번 범위에 포함하지 않는다. 이러한 요소는 기능적으로 의미가 있을 수 있으나, 현재 프로젝트의 목적과 일정 안에서는 핵심이 아니다. 본 문서는 기능을 무분별하게 확장하기보다, 데이터 생성에 직접 기여하는 핵심 흐름에 집중하는 것을 원칙으로 한다.

## 사용자가 수행해야 하는 핵심 흐름

본 프로토타입이 반드시 지원해야 하는 사용자 흐름은 다음과 같다. 사용자는 먼저 가입하거나 로그인한다. 이후 홈 화면에 진입하고, 카테고리 또는 상품 목록을 통해 상품을 탐색한다. 관심 있는 상품의 상세 정보를 확인한 뒤 장바구니에 상품을 추가한다. 장바구니 화면에서 현재 담긴 상품을 확인하고 필요 시 제거할 수 있어야 하며, 이후 체크아웃을 시작해 쿠폰을 적용하고 주문을 생성한다. 결제는 실제 PG와 연동하지 않고 성공 또는 실패를 시뮬레이션하는 방식으로 처리한다. 결제가 끝나면 주문 내역을 조회할 수 있어야 하며, 최종적으로 구매한 상품에 대해 리뷰를 작성할 수 있어야 한다.

이 사용자 흐름은 단순한 화면 이동 절차가 아니라, 운영계 데이터와 이벤트 로그를 생성하는 기준 시나리오이기도 하다. 따라서 각 단계는 사용자 입장에서 기능적으로 자연스러워야 할 뿐 아니라, 데이터 측면에서도 추적 가능해야 한다.

## 기능별 구현 기준

각 기능은 사용자 경험을 구성하는 화면 기능인 동시에, 운영계 데이터와 이벤트 로그를 생성하는 단위로 이해해야 한다. 따라서 기능 구현은 화면 동작만 만족하는 것으로 끝나지 않고, 어떤 테이블에 어떤 데이터가 기록되며 어떤 이벤트가 발생하는지까지 함께 고려해야 한다.

### 사용자 인증과 세션 관리

사용자 인증 기능은 계정 생성과 로그인이라는 기본 역할을 수행한다. 그러나 이 프로토타입에서 더 중요한 것은 사용자 식별과 세션 추적의 기반을 마련하는 일이다. 회원가입과 로그인은 `users` 테이블의 생성 및 갱신과 연결되어야 하며, 사용자가 서비스를 이용하기 시작하는 시점에는 `sessions` 데이터가 생성되어야 한다.

세션 관리는 이후 사용자 행동을 순서대로 복원하고, 특정 세션이 상품 조회, 장바구니 담기, 주문으로 어떻게 이어졌는지를 분석하는 데 필요하다. 따라서 인증 기능은 단순 계정 관리에 그치지 않고, 추후 퍼널 분석과 세션 기반 전환 분석의 출발점 역할을 해야 한다.

### 상품 탐색

상품 탐색 영역은 사용자가 구매 의도를 명확히 드러내기 이전의 행동을 담는다. 홈 화면, 카테고리 탐색, 상품 목록 조회, 상품 상세 조회가 이 영역에 포함된다. 이 기능들은 단순히 화면을 보여주는 수준이 아니라, 어떤 탐색 경로를 통해 사용자가 상품에 도달했는지에 대한 맥락을 남겨야 한다.

운영계 관점에서는 `categories`, `products`, `sessions`와 밀접하게 연결되며, 이벤트 관점에서는 `home_viewed`, `page_viewed`, `category_viewed`, `product_viewed`와 같은 행동 로그를 생성해야 한다. 이 영역은 이후 상품 상세 조회 대비 장바구니 전환율, 카테고리별 탐색 성과, 진입 경로별 행동 차이 등을 분석하는 기반이 된다.

### 장바구니 관리

장바구니 기능은 사용자의 탐색 행동이 구매 의도로 전환되는 첫 단계다. 사용자는 장바구니를 생성하고, 상품을 담고, 제거하고, 현재 상태를 조회할 수 있어야 한다. 이 기능은 `carts`와 `cart_items` 테이블에 직접적인 영향을 주며, 동시에 `cart_viewed`, `cart_item_added`, `cart_item_removed` 이벤트를 만들어야 한다.

장바구니는 커머스 분석에서 매우 중요한 신호다. 사용자가 어떤 상품에 관심을 보였는지, 어느 시점에 구매 의도를 드러냈는지, 어떤 상품이 자주 담겼다가 제거되는지 등의 해석은 이 데이터에서 시작된다. 따라서 장바구니 기능은 화면 완성도보다 상태 관리와 데이터 정합성이 우선되어야 한다.

### 체크아웃, 주문, 결제

체크아웃과 주문, 결제는 본 프로토타입의 핵심 비즈니스 흐름이다. 사용자는 체크아웃 단계에 진입해 주문 예정 상품을 확인하고, 필요 시 쿠폰을 적용한 뒤 주문을 생성할 수 있어야 한다. 결제는 실제 PG와 연결하지 않고 시뮬레이션으로 처리하지만, 성공과 실패 모두를 상태 변화로 남겨야 한다.

이 과정은 `coupons`, `orders`, `order_items`, `payments` 테이블에 데이터를 생성하거나 갱신하며, 이벤트 측면에서는 `checkout_started`, `coupon_applied`, `order_created`, `payment_succeeded`, `payment_failed`를 발생시켜야 한다. 특히 결제는 실제 금융 거래가 아니더라도 주문과 결제 상태가 데이터로 명확히 구분되어야 한다. 그래야 이후 주문 대비 결제 성공률, 결제 실패 패턴, 쿠폰 적용과 전환의 관계 같은 분석이 가능해진다.

### 리뷰 작성

리뷰 기능은 구매 이후의 사용자 행동을 다룬다. 사용자는 자신이 실제로 구매한 상품에 대해서만 리뷰를 작성할 수 있어야 하며, 이 기능은 `reviews`, `order_items`, `products`, `users`와 연결된다. 이벤트로는 `review_created`가 발생해야 한다.

리뷰 기능은 프로토타입 전체에서 우선순위가 가장 높은 기능은 아니지만, 구매 후 행동을 데이터로 남긴다는 점에서 의미가 있다. 이를 통해 구매 이후 참여율, 상품별 리뷰 생성률, 주문 상품과 후기의 연결 구조를 검증할 수 있다.

## 페이지 구성 원칙

프로토타입에서 제공되는 화면은 복잡할 필요는 없지만, 각 기능 흐름을 수행하기에는 충분해야 한다. 홈 화면은 사용자가 처음 진입해 탐색을 시작하는 지점이어야 하며, 로그인 및 회원가입 화면은 인증 기능을 수행해야 한다. 상품 목록 화면은 카테고리 기반 탐색과 상품 선택의 역할을 하고, 상품 상세 화면은 가격, 상품명, 브랜드, 장바구니 담기와 같은 기본 정보를 제공해야 한다.

장바구니 화면은 현재 담긴 상품을 보여주고 체크아웃으로 이어지는 역할을 해야 하며, 체크아웃 화면은 주문 예정 상품 확인, 쿠폰 적용, 주문 생성, 결제 시뮬레이션을 담당한다. 주문 내역 화면은 생성된 주문과 주문 상태를 확인할 수 있어야 하고, 리뷰 작성 화면은 구매한 상품에 대해 평점과 후기를 입력할 수 있어야 한다.

이들 화면은 화려할 필요는 없지만, 각 페이지가 어떤 기능적 책임을 가지는지는 명확해야 한다.

## API 범위

백엔드 API는 위 기능 흐름을 지원할 수 있을 만큼의 최소 범위를 가져야 한다. 인증 영역에서는 회원가입과 로그인 API가 필요하다. 상품 탐색 영역에서는 카테고리 조회, 상품 목록 조회, 상품 상세 조회 API가 있어야 한다. 장바구니 영역에서는 장바구니 생성, 조회, 장바구니 항목 추가 및 제거 API가 필요하다.

주문 및 결제 영역에서는 체크아웃 시작, 주문 생성, 결제 결과 시뮬레이션, 주문 내역 조회 API가 필요하며, 리뷰 영역에서는 리뷰 생성 API가 필요하다. 경로와 세부 명칭은 구현 중 일부 조정될 수 있지만, 기능적으로는 이 범위를 벗어나지 않아야 한다.

## 운영계 데이터 생성 기준

> [!IMPORTANT]
> 프로토타입은 실행 결과로 운영계 테이블에 실제 데이터를 생성해야 한다.  
> 시드 데이터와 런타임 생성 데이터를 구분해 관리하는 것이 기본 원칙이다.

이 프로토타입은 실행 결과로 운영계 데이터가 실제로 생성되어야 한다. 대상 테이블은 `users`, `sessions`, `categories`, `products`, `carts`, `cart_items`, `campaigns`, `coupons`, `orders`, `order_items`, `payments`, `reviews`다.

이 중 `categories`, `products`, `campaigns`, `coupons`는 초기 구동 시 시드 데이터로 적재하는 것이 적절하다. 반면 `users`, `sessions`, `carts`, `cart_items`, `orders`, `order_items`, `payments`, `reviews`는 실제 사용자 또는 테스트 흐름에 의해 런타임 중 생성되도록 해야 한다. 이렇게 구분하면 서비스 초기 상태를 빠르게 마련하면서도, 실제 행동 기반의 동적 데이터와 구분할 수 있다.

## 이벤트 로그 생성 기준

> [!WARNING]
> 이벤트 로그가 누락되면 이 프로젝트의 데이터 엔지니어링 목적 자체가 약해진다.  
> 운영계 기록과 이벤트 발생은 같은 기능 흐름 안에서 함께 설계되어야 한다.

이 프로토타입은 운영계 데이터만 생성하는 것으로 충분하지 않다. 사용자 행동과 비즈니스 상태 변화도 이벤트로 남겨야 한다. 사용자 행동 이벤트에는 홈 조회, 페이지 조회, 카테고리 조회, 상품 상세 조회, 장바구니 조회, 장바구니 담기, 장바구니 제거, 체크아웃 시작, 쿠폰 적용, 리뷰 생성이 포함된다. 비즈니스 상태 변화 이벤트에는 주문 생성과 결제 성공 및 실패가 포함된다.

이벤트 생성은 기능 구현 이후 덧붙이는 부가 기능이 아니라, 프로토타입의 본래 목적 중 하나다. 따라서 각 기능은 운영계 기록과 이벤트 발생이 같은 흐름 안에서 처리되어야 하며, 나중에 운영계 데이터와 이벤트 로그를 상호 검증할 수 있도록 식별자와 시각 정보가 일관되게 유지되어야 한다.

## 비기능적 제약과 구현 원칙

> [!CAUTION]
> UI 완성도나 상용 수준의 복잡성을 우선하면 범위와 일정이 쉽게 무너질 수 있다.  
> 본 프로토타입은 빠른 구현과 신뢰 가능한 데이터 생성에 초점을 둔다.

본 프로토타입은 기능 흐름의 완결성과 데이터 생성 구조를 우선한다. UI 완성도, 상용 수준의 예외 처리, 고도화된 운영 기능은 우선순위가 아니다. 따라서 구현은 빠르고 단순해야 하지만, 데이터는 신뢰할 수 있어야 한다.

결제는 반드시 시뮬레이션 방식으로 처리해야 하며 실제 외부 결제 시스템과는 연동하지 않는다. 이를 통해 개발 복잡도를 줄이면서도 결제 성공과 실패라는 중요한 비즈니스 상태를 재현할 수 있다. 개발 환경은 Frontend에 Next.js와 TypeScript, Backend에 FastAPI와 Python, Database에 PostgreSQL을 사용하는 것을 전제로 하며, 로컬 환경은 Docker Compose를 기준으로 한다.

## Sprint 1 우선 구현 범위

> [!TIP]
> Sprint 1은 전체 완성보다 기반을 다지는 구간으로 보는 것이 적절하다.  
> 화면 고도화보다 데이터 구조와 핵심 API를 먼저 안정화하는 편이 효율적이다.

Sprint 1에서는 전체 서비스 완성보다 기반 구조를 만드는 데 집중해야 한다. 이 시점에서 우선 구현해야 하는 것은 기능 명세서 작성, 프론트엔드 화면 구조 설계, 운영계 DB 스키마 DDL 초안 작성, 시드 데이터 설계, 백엔드 초기 세팅, 회원가입과 로그인 API, 세션 생성 및 추적 로직, 카테고리 및 상품 목록/상세 조회 API, 장바구니 생성·추가·조회 API, 초기 주문 생성 API다.

이 범위는 이후 프론트엔드 연결과 이벤트 로그 정교화, 통합 검증을 위한 토대를 마련하는 데 목적이 있다. 따라서 Sprint 1에서는 UI 완성도보다 데이터 구조와 핵심 API의 안정성을 우선해야 한다.

## Sprint 1 이후 확장 방향

기반이 안정화된 이후에는 프론트엔드 페이지 구현을 본격화하고, 체크아웃 흐름과 결제 시뮬레이션을 더 정교하게 만들 수 있다. 또한 리뷰 작성 기능 연결, 이벤트 로그 발생 구조 고도화, 운영계 데이터와 이벤트 로그의 정합성 점검, 데이터 파이프라인 연계를 위한 준비 작업도 이 시점부터 본격적으로 진행할 수 있다.

Sprint 1 이후의 작업은 기능을 새로 늘리는 차원이라기보다, 이미 정의된 기능과 데이터 흐름을 사용자 화면과 로그 구조에 안정적으로 연결하는 단계로 이해하는 것이 적절하다.

## 완료 판단 기준

> [!IMPORTANT]
> 이 프로토타입의 성공 기준은 서비스 외형이 아니라 데이터 생성 환경의 신뢰성에 있다.  
> 핵심 흐름 수행과 운영계·이벤트 데이터의 일관된 생성이 완료 판단의 기준이다.

이 프로토타입은 겉보기에 완성도 높은 커머스 웹사이트처럼 보일 때 성공하는 것이 아니다. 사용자가 가입 또는 로그인 이후 상품을 탐색하고 장바구니를 거쳐 주문을 생성할 수 있어야 하며, 그 과정에서 운영계 테이블에 데이터가 일관되게 쌓이고, 핵심 사용자 행동 이벤트와 주문·결제 상태 변화 이벤트가 생성되어야 한다. 그리고 그렇게 생성된 데이터가 이후 데이터 파이프라인, 품질 검증, 모니터링, BI 설계의 입력값으로 활용 가능해야 한다.

결국 이 문서가 정의하는 성공 기준은 서비스의 외형이 아니라, 신뢰 가능한 데이터 생성 환경을 구축했는가에 있다.
